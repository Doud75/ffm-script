# Contributing to ffm-script

Thanks for your interest in improving `ffm-script`! This document explains how to
set up the project, the workflow we follow, and what CI expects before a pull
request can be merged.

## Getting started

You'll need **Node.js >= 22** and **FFmpeg** (which provides both `ffmpeg` and
`ffprobe`). The install instructions per platform are in the
[README](README.md#prerequisites); if the binaries aren't on your `PATH`, point the
library at them with the `FFMPEG_PATH` and `FFPROBE_PATH` environment variables.

This project uses [pnpm](https://pnpm.io/) (pinned via `packageManager`), managed by
Corepack:

```sh
git clone https://github.com/Doud75/ffm-script.git
cd ffm-script
corepack enable
pnpm install --frozen-lockfile
```

## Project layout

| Path                        | What lives there                                          |
| --------------------------- | --------------------------------------------------------- |
| `src/`                      | Library source (the public API and its internals)         |
| `__tests__/`                | Jest test suites                                           |
| `fixtures/`                 | Sample media files used by the tests                      |
| `skills/ffm-script/`        | The shared AI Agent Skill (`SKILL.md`)                    |
| `dist/`                     | Build output — generated by `pnpm build`, never edited by hand |

## Development workflow

1. Branch off `main`. We name branches after the change: `feat/<name>` for new
   features, `refact/<name>` for refactors (e.g. `feat/stream`, `refact/testCodecov`).
2. Make your change in `src/`.
3. Add or adjust tests in `__tests__/`. New behavior should be covered.
4. If you changed the **public API**, keep the docs in sync:
   - update `README.md`,
   - add an entry to `CHANGELOG.md` under the next version heading (we follow
     [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) + [SemVer](https://semver.org/)),
   - update the Agent Skill at `skills/ffm-script/SKILL.md` so it stays accurate.

## Before you push (must pass CI)

CI runs lint, formatting, typecheck, tests with coverage, and a build — all must be
green. Run the same checks locally first:

```sh
pnpm lint            # ESLint
pnpm format:check    # Prettier (use `pnpm format` to auto-fix)
pnpm typecheck       # tsc --noEmit
pnpm test:coverage   # Jest with coverage — keep coverage above 90%
pnpm build           # tsup (ESM + CJS)
```

## Commit messages

Commits follow a simple `type : message` style (note the space before the colon):

```
feat : parallel convert executor
fix : point skill test at skills/ffm-script/SKILL.md
refacto : test to codecov > 90
```

Common types: `feat`, `fix`, `refacto`, `docs`.

## Opening a pull request

- Target the `main` branch, and keep one feature per PR.
- Fill in the pull request template and link the related issue (`Closes #123`).
- Make sure CI is green.
- **Do not bump the version** in `package.json` — releases are handled by the
  maintainer.

## Releases

Publishing is maintainer-only and fully automated: pushing a `v*` tag triggers the
release workflow, which publishes to npm via OIDC Trusted Publishing and creates the
matching GitHub Release. The package is never published from a local machine, and
contributors don't need to touch the version.
